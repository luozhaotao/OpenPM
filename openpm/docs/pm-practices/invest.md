# INVEST 原则

> 来源：*INVEST in Good Stories* — Bill Wake (2003)
> 本文档包含 INVEST 助记符的 6 维度完整定义、检查清单和在 OpenPM 中的应用。

---

## 1. INVEST 助记符

INVEST 是评估用户故事（User Story）质量的 6 维度检查清单。一个好的用户故事应该是：

| 字母 | 维度 | 含义 |
|------|------|------|
| **I** | Independent | 独立的 |
| **N** | Negotiable | 可协商的 |
| **V** | Valuable | 有价值的 |
| **E** | Estimable | 可估算的 |
| **S** | Small | 小的 |
| **T** | Testable | 可测试的 |

---

## 2. 各维度详解

### 2.1 I — Independent（独立）

故事应该尽可能独立。相互依赖的故事会使优先级排序、估算和规划变得困难。

**检查清单：**
- 这个任务是否可以独立开发和测试，不需要等待其他任务完成？
- 如果两个任务有顺序依赖，是否可以合并为一个，或重新拆分以消除依赖？
- 任务的 `depends_on` 是否超过 2 个？（超过则说明拆分不合理）

**反模式：**
```
task-001: 搭建认证框架
task-002: 实现登录（依赖 task-001）
task-003: 实现注册（依赖 task-001）
task-004: 实现密码找回（依赖 task-002、task-003）
```

**改进（消解依赖）：**
```
task-001: 实现邮箱+密码登录（含表单联调，无需框架先行）
task-002: 实现注册（含表单，与 task-001 独立）
task-003: 实现密码找回（通过邮件，与注册/登录独立）
```

### 2.2 N — Negotiable（可协商的）

故事不是合同，而是对功能的简短描述，细节可以通过沟通协商确定。

**检查清单：**
- 故事标题是否只描述了"做什么"，没有预设"怎么做"？
- 验收标准是否聚焦于可观察的行为，而非实现细节？
- 如果实现方式发生变化，故事描述是否需要大幅修改？

**反模式：**
```
title: "用 Redis 缓存用户会话"
（预设了实现技术）
```

**改进：**
```
title: "用户登录后会话保持 30 分钟"
（描述行为，不预设技术）
```

### 2.3 V — Valuable（有价值的）

故事必须对用户或利益相关者提供可感知的价值。

**检查清单：**
- 完成这个任务后，用户/系统能否感知到差异？
- 这是否是纯技术任务（如"重构"、"搭建项目结构"）？如是，考虑它是否应该成为一个独立 task，还是作为其他 task 的一部分
- 如果去掉这个任务，是否会影响用户可感知的功能？

**注意：** 技术债务和基础设施任务仍有价值，但应关联到具体的用户价值。如"重构认证模块以提高登录速度"连接了重构与用户价值。

### 2.4 E — Estimable（可估算的）

故事要有足够的信息，使得团队能够对工作量做出合理估算。

**检查清单：**
- 是否了解需要做什么？（缺少足够信息则不可估算）
- 是否了解需要修改哪些代码/文件？
- 是否有类似的任务可以参考其工作量？
- 如果"太大而无法估算"，说明需要进一步拆分

**反模式：**
```
title: "实现用户系统"
（太大，无法估算。涵盖注册、登录、找回、权限……）
```

**改进——拆分为多个可估算的任务：**
```
task-001: 实现邮箱注册功能
task-002: 实现邮箱+密码登录
task-003: 实现密码找回
task-004: 实现角色权限基础框架
```

### 2.5 S — Small（小的）

故事应该在合理的努力范围内，最好能在**一个 Sprint 内完成**。对 OpenPM 而言，一个任务应在**1 个工作日**内完成。

**检查清单：**
- 这个任务能否在 1 天内完成？
- 如果不能，哪些验收标准可以从这个任务中移出，形成新任务？
- 是否包含太多的验收标准？（>5 条 AC 通常是任务过大的信号）

**拆分技巧：**
| 拆分维度 | 示例 |
|----------|------|
| 按数据变体 | "注册" 拆为 "邮箱注册" + "手机号注册" |
| 按操作 | "管理用户" 拆为 "创建用户" + "编辑用户" + "删除用户" |
| 按界面 | "用户首页" 拆为 "导航栏" + "信息卡片" + "快捷操作" |
| 按场景 | "登录" 拆为 "正常登录" + "错误处理" + "记住密码" |
| 按质量属性 | 先实现基本功能，性能和安全性作为独立 task |

### 2.6 T — Testable（可测试的）

故事必须有清晰、客观的验证方式。好的验收标准让任何人都能判断故事是否完成。

**检查清单：**
- 每一条验收标准是否可以客观验证（是/否）？
- 验收标准是否使用了具体的、可观察的描述，而非模糊的形容词？
- 如果无法测试，是否缺少明确的标准？

**反模式：**
```
ac:
  - "登录体验良好"
  - "系统响应快"
  - "界面友好"
```
（主观、不可测试）

**改进：**
```
ac:
  - "输入正确邮箱和密码后，点击登录按钮，3 秒内跳转到首页"
  - "输入错误密码时，显示红色提示'密码错误'，输入框不重置"
  - "连续 5 次输入错误密码后，账户锁定 15 分钟并显示提示"
```
（客观、可验证）

## 3. INVEST 记分卡

在创建 Task 时，对每条 AC 按 INVEST 6 维度打分（0-1 分/维），总分 ≥ 4 分方可进入 Sprint Backlog。

| 维度 | 评分标准 |
|------|---------|
| I - 独立 | 0 依赖 = 1 分；1 依赖 = 0.5 分；≥2 依赖 = 0 分 |
| N - 可协商 | 有 AC 且不预设技术 = 1 分；AC 含技术实现细节 = 0.5 分；无 AC = 0 分 |
| V - 有价值 | 用户可感知 = 1 分；纯技术任务但有用户影响 = 0.5 分；纯技术无用户影响 = 0 分 |
| E - 可估算 | 明确范围和改动文件 = 1 分；大致了解 = 0.5 分；无法判断 = 0 分 |
| S - 小 | ≤1 天 = 1 分；≤2 天 = 0.5 分；>2 天 = 0 分 |
| T - 可测试 | 每条 AC 可客观验证 = 1 分；部分 AC 主观 = 0.5 分；无 AC = 0 分 |
